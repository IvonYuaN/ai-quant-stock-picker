# Short-Term Realtime Roadmap

本文件是 2026-06-30 起的当前执行地图，用来把项目从“泛化量化选股仓库”收束为“短线实时决策研究系统”。

它不替代 [architecture.md](/Users/ivon/Documents/AI量化选股/docs/architecture.md) 的模块契约；它只回答三件事：

1. 现在最优先的目标是什么
2. 哪些能力属于盘中实时链路，哪些只允许留在历史验证链路
3. 后续 agent/default workflow 应该沿哪几条并行主线推进，避免再次跑偏

## 背景收敛

基于当前仓库状态与既有协作记录，项目方向已经稳定到以下共识：

- 系统只辅助人工研究和纸面决策，不自动下单
- 短线/超短线优先，盘中实时新鲜度高于历史覆盖广度
- 历史数据仍然重要，但用途严格限定为回测、walk-forward、阈值冻结、agent 评估
- 国内外高价值信息源会影响建议质量，但只能作为上下文和增强层
- 多 agent 讨论是“建议增强与质疑机制”，不是交易指令引擎

## 唯一目标

将 AQSP 演进为面向短线决策研究的实时多源数据与多代理研判系统：

- 实时数据优先于历史数据
- 历史数据仅用于回测与验证
- 策略围绕短线机会检测与受控自动优化
- 整合国内外高价值信息源进入决策流程
- 构建多个决策 agent 的讨论与汇总机制
- 不触碰自动下单边界

## 交付目标约束

- 并行优先：主链路遇到长任务时，立即把互不重叠的审查、测试、文档和旁路修复交给独立子任务，主线继续推进，不以等待单个任务作为默认流程。
- 用户无感：任何开发、测试、抓取或部署任务必须后台运行，有锁、日志、超时和退出路径，不得占用用户前台应用、浏览器或生产主链资源。
- 闭环交付：代码通过本地验证不等于上线；每个影响运行效果的改动都必须经过提交、服务器部署、health、数据契约和用户可见结果检查，未完成验证只能标记为未交付。
- 失败可见：遇到数据不足、门禁失败、部署异常或旧进程残留时，停止扩展功能并报告事实、影响和恢复动作，不用话术掩盖半成品。

## 工作边界

### 盘中实时链路

这些能力默认服务于 `live_short`：

- 实时行情、分时 bar、盘口 quote
- 数据源 fallback 与 freshness guard
- 盘中候选、观察、阻塞、纸面执行提示
- 消息面催化、市场情绪、北向/融资等上下文增强
- 多 agent 辩论与组合汇总

要求：

- 如果实时源不适配短线链路，应直接 fail loud
- 盘中任务不能把纯历史源伪装成实时源
- 历史统计不能以未来视角污染盘中建议

### 历史验证链路

这些能力默认服务于 `walkforward` / `pit`：

- sqlite/raw 历史库
- walk-forward / CSCV / DSR / PBO
- PIT 财务、公告披露日、指数成分回放
- 自动优化、参数提案、因子评估
- 多 agent 复盘评估和角色表现跟踪

要求：

- 不复权原始价格和 point-in-time 披露口径是硬约束
- 历史验证结果只能进入阈值/权重/报告，不直接覆盖盘中 deterministic score

## 四条并行主线

### 1. 实时数据主线

目标：把 `live_short` 真正做成“实时优先、降级明确、失败可见”。

当前关注：

- `intraday` / `realtime` / `fetch_daily` 的职责切分
- 实时源与历史源适配校验
- 盘中任务输出与正式收盘 ledger 隔离

### 2. 历史验证主线

目标：把历史数据只用于“证据生产”，不再和盘中路径混线。

当前关注：

- walkforward 入口只走允许的历史源
- PIT / 披露日 / 原始价约束持续守住
- 自动优化只产出 proposal，不直接生效

### 3. 信息融合主线

目标：把国内外高价值信息源接入成“决策上下文层”，而不是噪音堆叠。

优先信息面：

- A 股个股/板块/指数新闻催化
- 国内政策、监管、行业景气度
- 海外指数、宏观风险偏好、美元/美债/商品等外部风险信号
- 北向资金、融资融券、市场情绪

要求：

- 所有信息源必须标注时效、来源和可操作性
- 全局新闻不能直接改写个股 score，只能增强建议解释或风控提示
- 已识别的跨市场传导线索可进入候选优先级增强层，但必须走确定性规则与配置化阈值，不能直接替代基础策略分

### 4. 多 Agent 研判主线

目标：把多 agent 从“展示功能”收束成真正有边界的建议增强层。

当前关注：

- agent 角色职责固定
- runtime 默认讨论视角覆盖技术、基本面、风控、板块、跨市传导、政策、融资、北向、情绪 9 个角色
- runtime 支持用 `AQSP_DEBATE_ROLES` / `AQSP_DEBATE_FOCUS_ROLES` / `AQSP_DEBATE_DISABLED_ROLES` 做场景化裁剪
- `AQSP_DEBATE_FOCUS_ROLES` 只负责把重点视角排到前面，不再默认把其他委员裁掉；真正停用统一走 `AQSP_DEBATE_DISABLED_ROLES`
- 未显式指定 `AQSP_DEBATE_ROLES` 时，盘前/盘中/收盘链路会按 `AQSP_RUN_TASK_ID` 自动选默认角色集
- 投票、加减权和汇总文案可解释
- agent 输出不能覆盖 deterministic score
- agent 历史表现只用于研究评估和角色再配置

前端表达约束：

- 首页、通知、候选复盘、执行卡片统一先出“跨市主线 | 先看 | 确认 | 失效”
- 同日 digest 永远先给结论，再展开多 agent 过程
- 首页讨论区先看委员会结果，再看讨论过程；同日速读不重复堆讨论明细
- 候选复盘顶部只放委员会结果，讨论过程下沉到“多 Agent 过程与证据”抽屉
- 多 agent 讨论用于压缩判断负担，不允许把用户重新推回大段原始信息流
- 首页 action rail 和先看顺序优先使用“当前结论 / 下一步 / 当前卡点 / 收盘后回看”这类操盘前台表达，避免展示研究流程术语

## 默认执行顺序

后续工作默认按以下优先级推进：

1. 先修实时链路和实时/历史边界
2. 再扩消息面和国内外信息融合
3. 再收敛多 agent 汇总与角色治理
4. 最后才把自动优化产物谨慎接回 runtime 参数层

## 不再做的偏航

- 不把一般性长线研究当作当前主线
- 不把历史覆盖率误当成盘中可用性
- 不把 LLM/agent 输出写成最终买卖结论
- 不为追求“看起来智能”牺牲数据新鲜度和可成交性
- 不在没有 guardrail 的情况下把自动优化结果直接写回运行参数
